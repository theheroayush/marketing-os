## 2024-05-18 - [HIGH] Fix Stored XSS via HTML Escaping
**Vulnerability:** User inputs like profile names (`p.name`) and team member emails (`email`) were being injected directly into the DOM via `innerHTML` without sanitization. This allows for Stored Cross-Site Scripting (XSS) if malicious payloads are saved in local storage.
**Learning:** In vanilla JavaScript applications utilizing template literals with `innerHTML` for rendering, failing to sanitize untrusted user data exposes the application to serious XSS vulnerabilities.
**Prevention:** Always implement and enforce a global HTML escaping utility function (e.g., `app.escapeHtml`) to sanitize any dynamic, user-controlled data before it is rendered to the DOM using `innerHTML`.
## 2024-05-15 - [XSS in Dynamic innerHTML Template Literals]
**Vulnerability:** Several dynamic string interpolations inside HTML template strings assigned to `.innerHTML` were injecting raw variables without escaping them, introducing Stored and Reflected XSS vulnerabilities (e.g. `searchQ`, `projectName`, and truncated message contents).
**Learning:** Even when string manipulation functions (like `.replace()` or `.slice()`) are used, if the result might contain unescaped user input or stored configurations, it can still trigger XSS when assigned via `.innerHTML`. Truncation (like `.slice(0, 120)`) does not neutralize HTML entities, it can even malform them, but tags can still execute if the payload fits or doesn't need to be closed properly.
**Prevention:** Always explicitly wrap any dynamically interpolated variables from `localStorage` or user input in `app.escapeHtml()` or a similar robust escaping function before inserting them into a template literal assigned to `.innerHTML`.
