## 2024-05-18 - [HIGH] Fix Stored XSS via HTML Escaping
**Vulnerability:** User inputs like profile names (`p.name`) and team member emails (`email`) were being injected directly into the DOM via `innerHTML` without sanitization. This allows for Stored Cross-Site Scripting (XSS) if malicious payloads are saved in local storage.
**Learning:** In vanilla JavaScript applications utilizing template literals with `innerHTML` for rendering, failing to sanitize untrusted user data exposes the application to serious XSS vulnerabilities.
**Prevention:** Always implement and enforce a global HTML escaping utility function (e.g., `app.escapeHtml`) to sanitize any dynamic, user-controlled data before it is rendered to the DOM using `innerHTML`.
## 2024-05-24 - [Fix XSS by escaping truncated regex-stripped text and UI strings]
**Vulnerability:** User inputs like search queries (`searchQ`), profile names (`projectName`), and regex-stripped chat histories (`s.messages...`) were injected directly into the DOM via `innerHTML` without final sanitization. Stripping tags with regex (`replace(/<[^>]*>/g, '')`) before truncating with `.slice()` can still leave unescaped text if malformed tags exist.
**Learning:** In vanilla JS apps using template literals with `innerHTML`, even seemingly harmless UI fields (like search boxes) or strings stripped of HTML via regex must be passed through a global HTML escaping function (like `app.escapeHtml`). Moreover, string manipulation like `.slice()` should occur *before* HTML escaping to avoid truncating entities (e.g., `&am...`).
**Prevention:** Consistently apply `app.escapeHtml()` to all dynamic data rendered via template literals, and perform truncation before escaping.
